Logic in Motion:
The Decision Engine

How does an autonomous agent decide whether to move an object or ignore it? We break down the Knowledge Base and the Inference Rules that drive intelligent management.

1. The Blueprint: Data Structures

Before making decisions, the system perceives the world through templates. The central entity is the Object (obj), and the desired result is a Task (task).

INPUT: (deftemplate obj)

📦
Attribute Type Description
id SYMBOL Unique Identifier
onto STRING Category (e.g., "apple")
pickable SYMBOL Can it be picked? (yes/no)
cur_room SYMBOL Current Location
dst_room SYMBOL Target Destination

OUTPUT: (deftemplate task)

⚡

When a rule fires, it generates a Task that tells the robot what to do.

ACTION: MOVE
Generated if the object is out of place.
ACTION: IGNORE
Generated if it is already correct or cannot be moved.
Example Output: (task (action move) (id obj1) (to kitchen) (reason "Out of place: bathroom -> kitchen"))

3. Detailed Code Analysis

A direct translation of the logic into the CLIPS language. Here we analyze each fragment of the provided knowledge base, along with its functional explanation.

(deftemplate obj
  (slot id (type SYMBOL))
  (slot onto (type STRING))
  (slot pickable (type SYMBOL))   ; yes/no
  (slot cur_room (type SYMBOL))   ; kitchen/bathroom/unknown
  (slot dst_room (type SYMBOL))   ; kitchen/bathroom/none
)

(deftemplate task
  (slot action (type SYMBOL))     ; move/ignore
  (slot id (type SYMBOL))
  (slot to (type SYMBOL))
  (slot reason (type STRING))
)

(deftemplate trace
  (slot msg (type STRING))
)

Template Definitions

The deftemplate instruction defines the structure of facts that will exist in working memory.

  • obj: Represents an object in the world. Defines its physical properties and location.
  • task: An order or command for the robot.
  • trace: Log messages used to debug actions.
(defrule not-pickable-ignore
  (obj (id ?id) (onto ?o) (pickable no))
  =>
  (assert (task (action ignore) (id ?id) (to none)
                (reason (str-cat "Not pickable according to ontology: " ?o))))
  (assert (trace (msg (str-cat "IGNORE " ?id " (not pickable, onto=" ?o ")"))))
)

Rule 1: Fixed Object

Checks whether the pickable attribute has the value no.

If it matches (the left-hand side or LHS of the rule), the rule fires. Variables ?id and ?o capture the object data to inject it into the actions (RHS), which assert (create) an ignore task and a trace log entry.

(defrule missing-destination-ignore
  (obj (id ?id) (onto ?o) (pickable yes) (dst_room none))
  =>
  (assert (task (action ignore) (id ?id) (to none)
                (reason (str-cat "Pickable but missing locatedIn/destination: " ?o))))
  (assert (trace (msg (str-cat "IGNORE " ?id " (no dst_room, onto=" ?o ")"))))
)

Rule 2: No Destination Configured

If the object can be picked (yes) but its destination is not defined or is none.

It generates the same output as Rule 1: an ignore task. It makes no sense to command the robot to move something if it does not know where to place it.

(defrule out-of-place-move
  (obj (id ?id) (onto ?o) (pickable yes) (cur_room ?cr) (dst_room ?dr))
  (test (neq ?cr unknown))
  (test (neq ?dr none))
  (test (neq ?cr ?dr))
  =>
  (assert (task (action move) (id ?id) (to ?dr)
                (reason (str-cat "Out of place: " ?cr " -> " ?dr " (onto=" ?o ")"))))
  (assert (trace (msg (str-cat "MOVE " ?id " " ?cr "->" ?dr " (onto=" ?o ")"))))
)

Rule 3: Object Out of Place

The main action rule. In addition to object attributes, it includes test to evaluate more complex logical conditions.

It ensures the current room is not unknown (neq ?cr unknown), that a valid destination exists (neq ?dr none), and the key check: the current room is different from the destination (neq ?cr ?dr). If satisfied, it asserts a move task.

(defrule already-correct-ignore
  (obj (id ?id) (onto ?o) (pickable yes) (cur_room ?cr) (dst_room ?dr))
  (test (or (eq ?cr ?dr) (eq ?cr unknown)))
  =>
  (assert (task (action ignore) (id ?id) (to ?dr)
                (reason (str-cat "Already correct (or unknown cur_room): " ?o))))
)

Rule 4: Object in the Correct Place

Acts as a safety condition for cases that do not fall under "Out of Place".

The test uses an or: it fires the ignore task if the current room equals the destination (eq ?cr ?dr), OR if the object's current location is unknown (eq ?cr unknown). Note that this rule intentionally does not generate a "trace" message.

⚙️ Execution Analysis in the Video:

  • 1. Initial Failure (Banana): At the beginning, the user tries to load a "banana". The system ignores it because the banana category is not registered in the knowledge ontology.
  • 2. Environment Modification: In the source code (line 81), the configuration is changed so that the camera and the environment load a cup/mug (cup).
  • 3. CLIPS Inference: After relaunching the simulation, CLIPS detects the cup in the "bathroom". Rule 3 (out-of-place-move) fires immediately because the correct destination for a cup is the kitchen.
