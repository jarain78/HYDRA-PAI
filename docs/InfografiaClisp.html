Lógica en Movimiento:
El Motor de Decisiones

¿Cómo decide un agente autónomo si mover un objeto o ignorarlo? Desglosamos la Base de Conocimiento y las Reglas de Inferencia que impulsan la gestión inteligente.

1. El Plano: Estructuras de Datos

Antes de tomar decisiones, el sistema percibe el mundo a través de plantillas (templates). La entidad central es el Objeto (obj), y el resultado deseado es una Tarea (task).

ENTRADA: (deftemplate obj)

📦
Atributo Tipo Descripción
id SYMBOL Identificador Único
onto STRING Categoría (ej. "manzana")
pickable SYMBOL ¿Se puede recoger? (yes/no)
cur_room SYMBOL Ubicación Actual
dst_room SYMBOL Destino Objetivo

SALIDA: (deftemplate task)

⚡

Cuando una regla se activa ("fires"), genera una Tarea para decirle al robot qué hacer.

ACCIÓN: MOVE
Se genera si el objeto está fuera de lugar.
ACCIÓN: IGNORE
Se genera si está correcto o es inamovible.
Ejemplo de Salida: (task (action move) (id obj1) (to kitchen) (reason "Out of place: bathroom -> kitchen"))

3. Análisis Detallado del Código

Traducción directa de la lógica al lenguaje CLIPS. Aquí analizamos cada fragmento de la base de conocimiento aportada, con su explicación funcional.

(deftemplate obj
  (slot id (type SYMBOL))
  (slot onto (type STRING))
  (slot pickable (type SYMBOL))   ; yes/no
  (slot cur_room (type SYMBOL))   ; kitchen/bathroom/unknown
  (slot dst_room (type SYMBOL))   ; kitchen/bathroom/none
)

(deftemplate task
  (slot action (type SYMBOL))     ; move/ignore
  (slot id (type SYMBOL))
  (slot to (type SYMBOL))
  (slot reason (type STRING))
)

(deftemplate trace
  (slot msg (type STRING))
)

Definición de Templates

La instrucción deftemplate crea la estructura de los hechos (facts) que existirán en la memoria de trabajo.

  • obj: Representa un objeto en el mundo. Define sus propiedades físicas y su ubicación.
  • task: Orden o mandato para el robot.
  • trace: Mensajes de registro (log) que se usarán para depurar las acciones.
(defrule not-pickable-ignore
  (obj (id ?id) (onto ?o) (pickable no))
  =>
  (assert (task (action ignore) (id ?id) (to none)
                (reason (str-cat "Not pickable according to ontology: " ?o))))
  (assert (trace (msg (str-cat "IGNORE " ?id " (not pickable, onto=" ?o ")"))))
)

Regla 1: Objeto Fijo

Evalúa si el atributo pickable tiene valor no.

Si coincide (el lado izquierdo o LHS de la regla), la regla se dispara. Las variables ?id y ?o capturan los datos del objeto para inyectarlos en las acciones (RHS), las cuales assert (afirman/crean) una tarea de ignorar y un trazo de registro.

(defrule missing-destination-ignore
  (obj (id ?id) (onto ?o) (pickable yes) (dst_room none))
  =>
  (assert (task (action ignore) (id ?id) (to none)
                (reason (str-cat "Pickable but missing locatedIn/destination: " ?o))))
  (assert (trace (msg (str-cat "IGNORE " ?id " (no dst_room, onto=" ?o ")"))))
)

Regla 2: Sin Destino Configurado

Si el objeto SÍ se puede recoger (yes) pero su destino no está definido o es none.

Genera la misma salida que la regla 1: una tarea de ignorar. No tiene sentido mandar al robot a mover algo si no sabe dónde dejarlo.

(defrule out-of-place-move
  (obj (id ?id) (onto ?o) (pickable yes) (cur_room ?cr) (dst_room ?dr))
  (test (neq ?cr unknown))
  (test (neq ?dr none))
  (test (neq ?cr ?dr))
  =>
  (assert (task (action move) (id ?id) (to ?dr)
                (reason (str-cat "Out of place: " ?cr " -> " ?dr " (onto=" ?o ")"))))
  (assert (trace (msg (str-cat "MOVE " ?id " " ?cr "->" ?dr " (onto=" ?o ")"))))
)

Regla 3: Objeto Fuera de Lugar

La regla principal de acción. Además de los atributos del objeto, incluye test para evaluar condiciones lógicas más complejas.

Se asegura de que la habitación actual no sea desconocida (neq ?cr unknown), que tenga destino válido (neq ?dr none) y la validación clave: que la actual sea diferente al destino (neq ?cr ?dr). Si se cumple, afirma una tarea tipo move.

(defrule already-correct-ignore
  (obj (id ?id) (onto ?o) (pickable yes) (cur_room ?cr) (dst_room ?dr))
  (test (or (eq ?cr ?dr) (eq ?cr unknown)))
  =>
  (assert (task (action ignore) (id ?id) (to ?dr)
                (reason (str-cat "Already correct (or unknown cur_room): " ?o))))
)

Regla 4: Objeto en Lugar Correcto

Actúa como condición de seguridad para los casos que no entran en "Out of Place".

El test utiliza un or: dispara la tarea de ignorar si la habitación actual es igual a la de destino (eq ?cr ?dr), O si la ubicación actual del objeto se desconoce (eq ?cr unknown). Note que esta regla intencionadamente no genera un mensaje de "trace".

ycb_pubullet_unloab_demo.py - Ejecución

⚙️ Análisis de la Ejecución en el Video:

  • 1. Fallo Inicial (Banana): Al principio, el usuario intenta cargar una "banana". El sistema la ignora porque la categoría banana no se encuentra registrada en la ontología del conocimiento.
  • 2. Modificación del Entorno: En el código fuente (línea 81), se cambia la configuración para que la cámara y el entorno carguen un vaso/taza (cup).
  • 3. Inferencia de CLIPS: Una vez lanzada la simulación de nuevo, CLIPS detecta el vaso en el "cuarto de baño" (bathroom). La Regla 3 (out-of-place-move) se dispara inmediatamente porque el destino correcto de una taza es la cocina.
  • 4. Ejecución Física: Se genera la orden final (task (action move) (to kitchen) ...), lo que desencadena la rutina cinemática del robot para ir, agarrar el objeto y trasladarlo a la bandeja azul de destino.
